## 📋 目的

Awareness Vault のコンテンツ品質を継続的に保証し、すべてのユーザーが一貫性のある高品質な学習教材を利用できる環境を維持する。

---

## 🎯 品質保証（QA）の役割

### **定義**
品質保証とは、**計画的・継続的にコンテンツの品質基準を満たしていることを確認し、問題を早期に発見・解決すること**。

### **責務範囲**

| 領域 | 責務 | 頻度 |
|------|------|------|
| **文件構造** | Markdown形式の正確性、Frontmatter完全性 | 新規追加時 |
| **コンテンツ** | 要約、ポイント、文字起こしの実装 | 新規追加時 |
| **メタデータ** | タグ、カテゴリ、日付フォーマットの確認 | 新規追加時 |
| **リンク整合性** | 内部リンク、外部リンク、画像の有効性 | 月1回 |
| **統一性** | セクション名、命名規則、表記の統一 | 月1回 |
| **Git管理** | マージコンフリクト、重複ファイル、エンコーディング | 都度 |

## 🔍 品質チェック基準

### **レベル1：新規ファイル追加時（100%チェック）**

```
□ Frontmatter（YAML）が正しい形式か
  ├─ title フィールド: 存在・値が適切か
  ├─ date フィールド: YYYY-MM-DD形式か
  ├─ category フィールド: 存在・値が適切か
  ├─ tags フィールド: 5個以上か
  └─ source_url フィールド: 有効か

□ 要約セクションが存在するか
  ├─ 200字以上の実テキストか
  ├─ テンプレート文がないか
  └─ 内容が適切か

□ ポイント/キーポイントセクションが存在するか
  ├─ 7個以上のポイントがあるか
  ├─ プレースホルダーがないか
  └─ 各ポイントに説明があるか

□ 文字起こし/本文セクションが存在するか
  ├─ 200字以上か
  └─ 実質的な内容か

□ リンク・画像が有効か
  ├─ 内部リンク [[file]] が存在するか
  ├─ 外部リンク [text](url) が有効か
  └─ 画像 ![[image]] が有効か

□ ファイル名とタイトルが一致するか

□ 特殊文字が含まれていないか（エンコーディング問題）
```

### **レベル2：定期監査（月1回）**

```
□ 全ファイルの統一性チェック
  ├─ セクション名の表記が統一されているか
  ├─ タグの命名規則が統一されているか
  ├─ カテゴリ分類が正確か
  └─ メタデータ形式が統一されているか

□ リンク整合性チェック
  ├─ 内部リンクが有効か（リンク先が存在するか）
  ├─ 外部リンクが有効か（404エラーなし）
  └─ 画像が正しく表示されるか

□ Git管理チェック
  ├─ マージコンフリクトがないか
  ├─ 重複ファイルがないか
  ├─ エンコーディングが統一されているか
  └─ コミットメッセージが適切か

□ コンテンツ品質チェック
  ├─ プレースホルダーテキストが残っていないか
  ├─ 古い情報が更新されているか
  └─ リンク切れがないか
```

### **レベル3：四半期監査（3ヶ月ごと）**

```
□ 全体構造の見直し
  ├─ ディレクトリ構成が最適か
  ├─ カテゴリ分類が適切か
  └─ ナビゲーションが効率的か

□ コンテンツの完全性確認
  ├─ すべてのファイルが要件を満たしているか
  ├─ 平均コンテンツ量が適切か
  └─ タグの多様性が十分か

□ ユーザー対応度確認
  ├─ 検索可能性が高いか
  ├─ 関連コンテンツへの参照が充実しているか
  └─ ユーザーレビューに基づく改善が反映されているか
```

## 🔧 チェックプロセス

### **新規ファイル追加時**

```
1. ファイルが正しい位置に保存されているか確認
   └─ 対象ディレクトリ: 01_講義 配下のカテゴリごと

2. ファイル名とタイトルが一致するか確認
   ├─ 形式: [数字]_[タイトル].md
   └─ 特殊文字がないか確認

3. Frontmatter が完全か確認
   ├─ 必須フィールド: title, date, category, tags
   └─ フォーマットが正確か

4. コンテンツセクションがすべて存在するか確認
   ├─ ## 要約
   ├─ ## ポイント
   └─ ## 文字起こし（全文）

5. 品質基準をクリアしているか確認
   ├─ 要約: 200-300字
   ├─ ポイント: 7個以上
   ├─ プレースホルダー: 削除済み
   └─ リンク: 有効か

6. Git に追加・コミット
   └─ コミットメッセージ: 明確・簡潔

7. チェックリスト完了をマーク
   └─ 監査ログに記録
```

### **定期監査時（月1回）**

```
1. 全ファイルリストを取得
   └─ 新規追加・削除・更新されたファイルを確認

2. レベル2チェックを実行
   └─ 前月以降の変更ファイルを優先チェック

3. 問題があれば記録
   ├─ 優先度を付けて分類
   └─ 担当者をアサイン

4. 改善レポートを作成
   └─ 実施日時、発見問題、改善内容を記録

5. コミットして履歴に残す
   └─ 監査ログファイルを更新
```

## 📝 監査ログ

すべての品質確認を記録します：

### **フォーマット**

```markdown

### 2026-06-27（初回監査）
- **実施者**: Claude Code
- **監査レベル**: 包括監査
- **対象ファイル**: 104ファイル
- **チェック項目**: 5項目
- **合格率**: 82.5% → 100%（修正後）
- **発見問題**: 4件
- **解決状況**: 100% 解決
- **修正コミット**: a2d7d0e

### 2026-07-27（予定）
- **実施者**: [担当者]
- **監査レベル**: 月次監査
- **対象ファイル**: 新規追加分
- **結果**: [待機中]
```

## 👥 責務者とスケジュール

### **新規ファイル追加時**
- **責務者**: コンテンツ作成者
- **チェック**: 品質保証（QA）チーム
- **承認**: リード者
- **期間**: 追加から24時間以内

### **月次監査**
- **実施者**: 品質保証（QA）チーム
- **頻度**: 毎月末
- **所要時間**: 2～3時間
- **報告**: 監査ログに記録

### **四半期監査**
- **実施者**: 品質保証（QA）チーム + リード者
- **頻度**: 3ヶ月ごと
- **所要時間**: 4～6時間
- **報告**: 四半期レポート作成

## 🔨 修正プロセス

問題が発見された場合：

```
1. 問題の分類と優先度付け
   ├─ P0（緊急）: 24時間以内に修正
   ├─ P1（重要）: 1週間以内に修正
   └─ P2（軽微）: 1ヶ月以内に修正

2. 修正の実施
   ├─ 対象ファイルを特定
   ├─ 修正内容を計画
   └─ 実装・テスト

3. 修正の検証
   └─ 修正後のチェック実施

4. コミット & レポート
   ├─ 修正をコミット
   └─ 監査ログに記録
```
